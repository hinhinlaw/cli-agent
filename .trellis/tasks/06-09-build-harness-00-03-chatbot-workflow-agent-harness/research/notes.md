# 00-03 研究笔记：系统边界

## 核心结论

ChatBot 解决对话问题，Workflow 解决确定流程问题，Agent 解决动态决策问题，Harness 解决稳定托管问题。

Agent 不是更高级的默认选项，而是为不确定任务付出的复杂度成本。

## 问题链

```text
用户只是需要理解和表达
-> 用 ChatBot 管理对话就够了
-> 任务步骤已经确定，只需要自动执行
-> 用 Workflow 更可靠、更便宜、更好测试
-> 任务路径不确定，需要运行中判断下一步
-> 才引入 Agent，让模型参与动态决策
-> Agent 一旦接触真实环境，就会产生工具、权限、状态、副作用和验证问题
-> 模型提出的动作不能直接等于系统执行
-> 需要 Harness 承接模型外部的工程控制
```

## ChatBot

适用条件：问题主要发生在对话里。

职责：

- 管理消息历史。
- 调用模型生成回答。
- 支持解释、总结、改写、问答。

限制：ChatBot 可以给建议，但不能把建议当成已执行动作。

## Workflow

适用条件：步骤已经确定。

职责：

- 按固定流程执行。
- 用程序稳定调度节点。
- 在合适节点使用 LLM。

判断：如果流程已经能写死，就不要让模型每一步重新发明流程。

## Agent

适用条件：任务路径必须在运行中根据观察结果决定。

职责：

- 在 loop 中让模型动态判断下一步。
- 把判断转成 tool intent。
- 用 observation 继续驱动下一轮。

成本：引入动态决策后，系统会变得更难测试、更难控制、更难解释。

## Harness

适用条件：Agent 要稳定进入真实环境。

职责：

- 管理执行边界。
- 管理权限和状态。
- 记录日志和 trace。
- 支持恢复和验证。
- 支持治理和评估。

Harness 不是更聪明的 Agent，而是让 Agent 可控运行的外部系统。

## 边界判断

核心问题：不确定性在哪里？

```text
不确定性在表达 -> ChatBot
不确定性已被流程消化 -> Workflow
不确定性必须运行时处理 -> Agent
Agent 要真实托管 -> Harness
```

坏味道：系统把工程控制让渡给模型语言。

典型表现：

- 没有工具结果事件流。
- 没有区分建议和已执行动作。
- 没有验证步骤，却允许模型宣布完成。
- 系统失败后只能看最终回答，无法回放过程。

## 教学 Harness 的落地点

教学项目可以保留两个入口体现边界：

- `POST /api/prompt`：一次性调试入口。
- `/api/runs` 加事件流：更接近 Harness 运行入口。

固定流程适合写在 API 或测试里。只有“下一步取决于观察结果”时才需要 Agent Loop。
